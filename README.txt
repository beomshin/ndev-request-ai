1) 문제·대상 업무
(1) 누가 쓰는가
1차 사용자는 PG결제개발실에 개발요청서를 제출하는 사내 현업 담당자입니다.
영업, 대외사업실, 운영, 재무 등 비IT 직군이 여기에 해당합니다.
2차 사용자는 요청서를 받아 개발에 착수하는 PG결제개발실 개발자와 PL이며,
확장 사용자는 합병 후 ONE-IT 체계에서 PG 도메인 개발요청을 작성하고 접수하게 될 
VAN사업본부 현업과 VAN개발실입니다.

(2) 어떤 상황인가
2025년 10월 합병이 진행되며 온오프라인 결제 인프라, 표준 규격, 정산 체계가 통합되고 있습니다.
그 결과 현업이 마주하는 결제 도메인의 폭이 합병 이전보다 훨씬 넓어졌고,
PG 표준결제창 변경, VAN API 연동, 해외카드 결제처럼 한 부서의 지식만으로
설명되지 않는 요청이 일상이 되었습니다.

(3) 어떤 업무 문제인가

1. 현업에서 신규 지불수단 추가 요청 티켓 작성
2. 관련 된 내용을 수집하고, 미비한 내용을 보완 요청하며 정책을 정의하기 위한 N번의 회의 진행 
3. 개발 착수

2번 과정에서 개발 티켓 당 평균 3~5회의 핑퐁이 발생합니다.

원인은 현업은 PG 기술 용어(MID, 제공채널, 부분취소, 인증/승인/매입 등)를 자세히 알 수 없고,
개발실에서는 그 정보 없이 개발에 착수할 수 없기 때문입니다.

합병 이후 도메인이 넓어지면서 이 격차는 더 크게 벌어지게 되고, 
결국 신규 가맹점 오픈이나 이벤트 결제처럼 매출과 직결되는 과제의 착수 지연과 기회비용 손실로 이어집니다.

(4) 우리가 정의한 핵심 업무 과제
현업이 기존 양식대로 작성한 요청서의 빈칸과 모호한 표현을 IT가 한번에 알아듣는 한장의 표준 요청서로 보완·번역한다.
정량적 목표는 이전에 개발 요건 확인을 위해 주고받던 질의 3~5회를 0~1회로 줄이고, 착수 지연을 단축하는 것입니다.


2) AI 활용
본 서비스는 Google Gemini 2.5 Flash(Google GenAI SDK 1.58.0)를 두 번의 분리된 호출로 사용합니다.
어디에 어떻게 썼는지 호출 단위로 작성합니다.

(1) LLM 호출 1번 : 요청서 보완과 구조화
입력은 현업이 Wizard에서 작성한 9개 필드와 PG 도메인 지식 문서 링크입니다. 
처리 단계에서 Gemini가 Google Search 도구로 사내 도메인 문서를 실시간 열람하면서 현업이 비워 두었거나
모호하게 적은 항목을 정책 문서 기반으로 보완하고 PG 기술 명세로 번역합니다.
출력은 표준 요청서 v1.7(8섹션, 35필드)의 구조화된 JSON이며, 이를 Markdown으로 변환합니다. 
현업이 확정해서 입력한 값은 LLM 추론값을 결정론적으로 덮어쓰는 보정을 거칩니다.

(2) LLM 호출 2번 : 거래 시퀀스 다이어그램 생성
입력은 호출 1번이 만들어 낸 Markdown과 동일한 도메인 문서입니다.
Gemini가 가맹점에서 PG, 인증기관, 카드사로 이어지는 메시지 흐름을 추론해 draw.io 호환 mxGraph XML을 생성하고, 
JGraphX가 이를 2배 해상도 PNG로 렌더링합니다. 
XML 파싱이 실패하더라도 대체 PNG가 자동으로 생성되어 서비스는 무중단으로 유지됩니다.

(3) AI를 쓰지 않았다면 풀기 어려웠던 지점 네 가지

첫째, 현업이 작성한 요청서의 "빈칸과 모호한 표현"을 사내 정책 문서로 보완하는 일입니다. 
Req-Genie는 현업에게 한 줄만 받는 도구가 아닙니다.
기존 개발요청서에서 작성하던 항목을 그대로 Wizard 단계에서 입력받습니다.
다만 현업이 도메인 지식 부족으로 비워두거나, 추상적으로 적어 둔 항목에 대해 AI가 사내 PG 정책 문서를 직접 참조해
채널(PC, Mobile), 결제방식, 인증/승인/매입주체, 부분취소 정책, 최소금액 같은 기술 명세 항목을 보완하여 35개 필드를 완성합니다.
규칙 기반 파서나 단순 템플릿으로는 현업이 쓴 다양한 표현, 오탈자, 약어를 매번 정책 문서와 매칭해 줄 수 없습니다. 
현업 입력과 정책 문서를 동시에 읽고 의미적으로 연결해 빈칸을 채우는 작업은 LLM의 의미 해석 능력이 본질적으로 필요한 일입니다.

둘째, "추론 가능"과 "추론 불가"를 AI가 스스로 분리하는 자기인식 설계입니다.
정책 문서로 보완할 수 있는 항목과 끝내 추론이 불가능해 현업 재확인이 필요한 항목을 AI가 스스로 분리해, 후자만 모아 한 번에 묻습니다. 
이 분리는 결정트리나 룰엔진이 흉내내기 어렵고, 핑퐁 횟수를 0~1회로 줄이는 결정적 기능이 바로 여기에서 나옵니다.

셋째, 도메인 문서를 매 요청마다 새로 해석하는 일입니다. 
PG정책, 연동규격서 스펙, provider 연동규격은 계속 개정됩니다. 
템플릿이나 매크로 방식은 개정 즉시 다시 짜야 하지만, LLM과 Google Search의 조합은 문서만 갱신하면
그 다음 요청부터 곧바로 반영됩니다. 유지보수 비용이 본질적으로 작아집니다.

넷째, 거래 흐름 다이어그램 자동 생성입니다. 
요청서 텍스트를 읽고 참여 기관과 메시지 순서를 추론해 시퀀스 다이어그램 XML을 만드는 일은, 
사람이 매번 draw.io에서 30분에서 1시간을 들여 그리던 업무입니다. 
룰 기반으로는 취소나 환불 흐름 포함 여부처럼 케이스마다 달라지는 분기를 자동화하기 어렵습니다. 
LLM의 추론과 구조화 출력 능력이 정확히 맞는 작업입니다.


3) 완성된 결과물과 작동 근거

(1) 최종 산출물
사용자가 Wizard 단계(기본 6단계, 조건부 1단계 추가)를 입력한 뒤 생성 버튼을 누르면 개발 요청서 파일이 생성되어 화면상에서 보여집니다.
보여지는 파일은 MD 파일로 복사가 가능하며 별도 가공 없이 바로 사내 협업 문서 시스템 Share에 작성할 수 있습니다. 
함께 생성된 거래 흐름 시퀀스 다이어그램(PNG)도 같은 화면에 표시되며, 이미지 파일로 내려받아 Share나 회의자료에 첨부할 수 있습니다.

(2) 끝까지 실제로 작동하는 부분 (MVP 시연 가능)
첫째, Wizard UI가 기본 6단계 + 조건부 1단계(S7)로 구현되어 있습니다.
Slide 1은 기능 구분(기존 수정 또는 신규 개발)을 선택하고,
Slide 2는 카테고리(PG표준결제창, API, 해외결제, 기타)를 고릅니다.
Slide 3은 기본 정보(작성자, 부서, 서비스명, 추진 배경, 가맹점/원천사)를,
Slide 4는 목표 일정과 일정 근거를,
Slide 5는 문제점과 개선점, 발생 빈도, 경쟁사 정보, 예상 수익과 손해를 입력받습니다(신규 개발 유형일 땐 문제점·발생 빈도는 자동 생략).
Slide 6은 기존 수정 유형일 때 AS-IS/TO-BE(UI 또는 API)를, 신규 개발 유형일 때 자유 메모를 입력받습니다.
Slide 7은 신규 + PG표준결제창 + 카드 조건을 모두 만족할 때만 노출되며, 신규 지불수단 등록을 위한 공통 포맷(6섹션·34필드)을 수집합니다.

둘째, LLM 호출 1번(요청서 보완과 구조화)이 Gemini 2.5 Flash로 실제 동작합니다.
결과로 35개 필드의 Markdown이 산출되며, AI 추론 근거가 명시되고 현업 재확인이 필요한 질문 목록이 자동으로 생성됩니다.

셋째, LLM 호출 2번(시퀀스 다이어그램)도 Gemini 2.5 Flash로 실제 동작합니다.
mxGraph XML 생성 후 JGraphX가 PNG로 렌더링하며, 순번 라벨(1, 2, 3)과 AI 추론 메시지에는 "(가정)" 표기가 붙습니다.

(3) demo 폴더 자료가 무엇을 보여 주는지

1. 개발요청서 유형선택.png 
Wizard Step 1 진입 화면입니다. 현업이 가장 먼저 만나는 질문이 "기존 서비스 수정·개선이냐, 신규 서비스 개발이냐"라는 단 하나의 비즈니스 질문임을 보여 줍니다. 기술 용어를 한 번도 묻지 않는다는 설계 원칙이 진입점부터 그대로 적용되어 있다는 점의 증빙입니다.

2. 개발요청서 대분류 선택.png 
Wizard Step 2의 대분류·세부유형 선택 화면입니다. pg표준결제창을 고르면 그 아래에 카드·계좌이체·가상계좌·휴대폰결제·기타의 세부유형이 자동으로 펼쳐집니다. 분기 트리가 백엔드에서 GET /api/catalog/로 내려와 동적으로 구성된다는 점, 그리고 현업이 자기 도메인 언어("카드결제 추가")만 알아도 분류가 끝난다는 점을 보여 줍니다.

2. 개발요청서 대분류 세부 선택.png
같은 Step 2 화면에서 API 대분류를 골랐을 때 신규API생성·빌링·SMS링크결제·기타로 세부유형이 다르게 펼쳐지는 모습입니다. 카테고리 선택 결과에 따라 이후 단계의 입력 폼이 동적으로 바뀐다는 점, 그리고 분기 트리가 한 폼에 고정되지 않고 카테고리별로 살아 있다는 점의 증빙입니다.

3. 개발요청서 기본 배경 설정.png 
Wizard Step 3, 기본·배경 정보 입력 화면입니다. 작성자(김규리)·작성 부서(PG결제개발실)·가맹점 정보(NICE I&T)·원천사 정보(토스페이)·추진 배경(가맹점 영업을 위한 토스페이 정기결제 도입)을 현업이 직접 입력한 모습이며, 작성일은 자동으로 채워집니다. "확정 입력값은 LLM 추론값을 결정론적으로 덮어쓴다"는 안전장치가 이 단계에서 수집된 값에 적용된다는 점의 출발점입니다.

4. 개발 요청서 상세1.png 
Wizard Step 4, 상세 1/3 화면입니다. 개발 서비스명(토스페이 정기결제 도입)·목표 일정(2026-09-30)·목표 일정 근거(가맹점 프로모션 런칭일)를 입력합니다. 목표 일정 입력란이 YYYYMMDD 형식으로 적어도 YYYY-MM-DD로 자동 변환된다는 점, 즉 Zod 기반 입력 검증이 실시간으로 동작한다는 점을 보여 줍니다.

5. 개발요청서 상세2.png 
Wizard Step 5, 비즈니스 임팩트 2/3 화면입니다. 경쟁사 정보(경쟁사 A사 토스페이 정기결제 서비스 중)·예상 수익(월 결제액 10억 증가 예상)·예상 손해/리스크(가맹점 이탈)를 입력합니다. 기술이 아닌 비즈니스 언어로만 채워지는 단계라는 점, 그리고 이 값들이 최종 요청서의 "6섹션 기대 효과"에 그대로 반영된다는 점의 시작점입니다.

6. 개발요청서 상세3.png 
Wizard Step 6, 상세 3/3 화면입니다. "추가로 알려주시고 싶은 내용" 자유 입력란과 "잘 모름 / 추가 확인 필요" 체크박스가 핵심입니다. 현업이 "어떤 정책 정의가 필요한지 모름"이라고 명시적으로 표시하면, 해당 항목이 화면 하단의 "추가 확인 필요 항목"에 즉시 누적됩니다. Req-Genie의 핵심 설계인 "모르는 것을 모른다고 말한다"가 UI 차원에서부터 강제된다는 점의 증빙입니다.

7. 개발요청서 신규 지불수단 입력 Form.png
 Wizard Step 7, 신규 지불수단 등록 공통 포맷 화면입니다. 인증주체·승인주체·매입주체·연동 지불수단·서비스 제공 채널·통화종류·복합과세·제휴사 연동 Key(CPID, API 접근제어)·인증·승인(할인쿠폰, 할부, 분담무이자, 최소 결제금액, 타임아웃 설정 등)·취소(부분취소, 취소가능 기한, 망취소)·기타까지 33개 정책 항목을 섹션별로 입력받습니다. 현업이 채우지 않은 항목은 다음 단계에서 AI가 사내 PG 정책 문서를 참조해 보완한다는 점, 즉 "기존 양식대로 작성하되 빈칸은 AI가 메운다"는 본 서비스의 핵심 동작 방식을 가장 직접적으로 보여 주는 화면입니다.

8. 개발요청서 수행 결과.png

4) 기존 방식 대비 차별성
평가 AI가 이 분야의 기존 관행을 모를 수 있으므로, 비교 대상을 먼저 명시합니다.

(1) 비교 대상
방식 A는 워드나 엑셀 양식을 메일로 주고받는 방식입니다.
방식 B는 사내 협업 문서 시스템 Share에 템플릿 폼을 작성하는 방식으로, 현재 PG결제개발실의 기본 방식입니다.
방식 C는 IT 기획자나 시니어 개발자가 현업을 인터뷰한 뒤 대신 요청서를 작성하는 방식입니다.
방식 D는 ChatGPT나 사내 LLM 채팅창에 자유 질문 형태로 묻는 방식입니다.

(2) 비교 축별 차이
도메인 지식 주입 측면에서, 방식 A와 B는 도메인 지식이 전혀 주입되지 않고,
방식 C는 사람의 머릿속 지식에 의존하며, 방식 D는 일반 지식에 그칩니다.
Req-Genie는 PG 도메인 문서를 실시간으로 참조합니다.

산출물 표준화 측면에서, 방식 A는 자유 서식이라 편차가 크고, 방식 B는필드를 채우지만 깊이가 얕으며, 
방식 C는 작성자마다 결과가 다르고, 방식 D는 호출할 때마다 결과가 다릅니다. 
Req-Genie는 표준으로 정의된 문서를 기준으로 필드를 강제합니다.

시퀀스 다이어그램 측면에서, 방식 A, B, C는 모두 사람이 30분에서 1시간을 들여 수작업으로 그리고, 방식 D는 텍스트 설명만 가능합니다.
Req-Genie는 자동 생성하여 별도 작업 시간이 들지 않습니다.

추론 근거의 투명성 측면에서, 방식 A와 B는 해당 사항이 없고, 방식 C는 사람의 기억에 의존하며, 방식 D는 블랙박스에 가깝습니다. 
Req-Genie는 모든 추론 항목의 근거를 출처와 함께 명시합니다.

"모름"에 대한 자기인식 측면에서, 방식 A와 B는 해당 사항이 없고, 방식 C는 작성자 역량에 의존하며, 방식 D는 환각 위험이 큽니다. 
Req-Genie는 추론 불가 항목을 자동 분리해 한 번에 모아 묻습니다.

결과적으로 핑퐁 횟수는 방식 A에서 3~5회, B에서 2~4회, C에서 1~2회, D에서 2~4회가 발생하지만, 
Req-Genie는 0~1회를 목표로 합니다. 

착수 지연 역시 방식 A는 3~5일, B는 2~4일, C는 2~3일, D는 측정이 어렵지만,
Req-Genie는 당일에서 1일을 목표로 합니다.

(3) 왜 다른가 (차별성의 근거 세 가지)
첫째, 도메인 RAG가 사내 정책 문서에 결합되어 있습니다.
방식 D는 일반 지식만 알지만, Req-Genie는 PG 정책 12건, 스펙 5건, provider 3건, 요청서 사례 4건, catalog와 README 2건의
사내 도메인 문서를 Google Search 도구로 매 호출마다 직접 열람합니다. 
일반 LLM이 절대 모르는 "자사의 부분취소 정책"이나 "자사의 인증 주체 규약" 같은 항목까지 채워 줍니다.

둘째, 모르는 것을 모른다고 분리하는 자기인식 설계입니다. 
방식 D는 모를 때도 그럴듯하게 채워서(환각) 오히려 핑퐁을 늘릴 수 있지만, 
Req-Genie는 입력값(확정), 문서로 추론 가능, 추론 불가의 세 가지 분류를 강제하고,
추론 불가 항목만 재확인 질문 목록으로 모아 한 번에 묻습니다.
환각을 핑퐁이 아니라 한 번의 깔끔한 재확인으로 흡수합니다.

셋째, 텍스트와 다이어그램을 한 번에 산출하는 멀티모달 결과물입니다.
방식 A, B, C는 시퀀스 다이어그램을 별도 수작업으로 그려야 하지만, 
Req-Genie는 동일 요청 한 건에 대해 텍스트 요청서와 시퀀스를 한 묶음으로 산출합니다.
현업은 작성된 md파일을 그대로 IT, 회의자료, 사내 협업 문서 시스템 Share에 첨부하면 끝입니다.
