# [PG 공통] 결제 오류 코드 및 안내 문구 표준 가이드

개발팀은 결제 실패 시 원천사/PG사에서 내려오는 에러 코드를 가맹점 화면에 그대로 노출하지 않고, 고객 친화적인 문구로 치환하여 노출하는 스펙을 기획에 반영해야 합니다. AI는 현업 인터뷰 시 이 예외 처리 문구 기획 여부를 검증합니다.

## 5.1 주요 실패 코드 및 기획서 반영 예외 처리 룰

### 1. 에러코드: `ERR_4032` (부부분 취소 금액 한도 초과 / 부분 취소 불가 거래)
- **의미:** 원천사 수수료 정책이나 수단 특성상 금액의 일부만 취소하는 것이 제한된 상태입니다.
- **가맹점 노출 권장 문구:** "부분 취소가 불가능한 거래입니다. 전체 취소 후 재결제 해주세요."
- **개발팀 예외 처리 룰:** 백엔드 환불 모듈에서 부분취소 가능 여부 플래그(`isPartialCancelable`)를 사전 검증하는 인터페이스를 구현해야 함을 기획에 반영하도록 가이드.

### 2. 에러코드: `ERR_LIMIT` (카드사 한도 초과 / 잔액 부족)
- **의미:** 사용자의 카드 한도나 머니 잔액이 부족하여 결제가 승인되지 않은 상태입니다.
- **가맹점 노출 권장 문구:** "결제 금액 한도가 초과되었습니다. 다른 결제 수단을 이용해 주세요."
- **개발팀 예외 처리 룰:** 결제창 이탈 시 주문서가 완전히 파괴되지 않고, 기존 주문 데이터를 유지한 채 결제 수단 선택창(주문서)으로 자연스럽게 리다이렉트 시키는 복귀 동선 확보 여부 검증.

### 3. 에러코드: `ERR_TIMEOUT` (제휴 원천사 응답 타임아웃)
- **의미:** 가맹점 서버가 최종 승인 API를 호출했으나 원천사 서버가 제한 시간 내에 응답하지 않은 경우입니다.
- **가맹점 노출 권장 문구:** "결제 처리 중 시간이 초과되었습니다. 주문 내역을 확인해 주세요."
- **개발팀 예외 처리 룰:** 시스템이 사용자에게 실패 팝업을 띄우기 직전, 백엔드에서 **망취소(NetCancel) API**를 즉시 자동 호출하도록 설계를 강제함. 거래 대사 원인 기록 로그 생성 스펙 확인.

### 4. 에러코드: `ERR_ALREADY` (중복된 주문 번호 인입)
- **의미:** 동일한 가맹점 주문번호(`Moid` / `orderId`)로 결제 승인이 연속으로 인입된 경우입니다.
- **가맹점 노출 권장 문구:** "이미 처리 중인 주문입니다. 잠시 후 다시 시도해 주세요."
- **개발팀 예외 처리 룰:** 클라이언트 UI 단에서 사용자가 '결제하기' 버튼을 더블 클릭(중복 클릭)하거나 새로고침을 할 수 없도록 버튼 비활성화(Debounce/Loading 스피너) 스펙이 UI 기획서에 포함되어 있는지 검증.
