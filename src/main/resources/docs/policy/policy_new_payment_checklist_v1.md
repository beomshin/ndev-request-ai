## 1. 정책 목적 및 적용 범위

본 정책은 **신규 지불수단 연동(NEW)** 요청서가 PG개발실에서 개발 가능한 범위인지 판별(R&R)하고, 기획서(요구사항 정의서)에 **반드시 정의되어야 할 22개 항목의 누락을 사전에 보완**하기 위한 거버넌스 기준이다. AI 어시스트는 현업이 요청서를 최종 제출하기 전, 대화를 통해 누락 항목을 역추적하여 보완한다(F8). 개발 대상이 아닌 요청은 즉시 반려한다(F1).

## 2. 사전 필수 정책 체크리스트 (22항목)

신규 지불수단 연동 시 아래 22개 항목이 기획서에 명시적으로 정의되어야 한다. 누락 시 AI는 해당 항목을 추가확인 항목으로 승격해 보완을 요구한다(`E-GOV-010`).

## 3. R&R 자동 반려 규칙

현업 인터뷰 중 아래 케이스 발견 시, AI는 요구사항 정의서를 생성하지 않고 **"PG개발실에서 개발하는 부분이 아닙니다"** 안내와 함께 즉시 대화를 종료(반려)한다.

| 규칙 | 케이스 | 안내(이관처) | 코드 |
|---|---|---|---|
| RNR-01 | 디자인/단순 UI 퍼블리싱 (버튼 색상·CSS·레이아웃 변경) | 디자인팀/퍼블리싱 파트 | E-GOV-001 |
| RNR-02 | 원천사/제휴사 자체 시스템 (외부 플랫폼 자체 화면 개발) | 제휴 마케팅팀 → 원천사 요청 | E-GOV-002 |
| RNR-03 | 단순 설정/운영 업무 (MID 비밀번호 초기화, 정산 데이터 추출) | 결제 운영/정산 파트 | E-GOV-003 |

## 4. 누락 항목 역추적 동작 (F8 연계)

```
1. funcType == 'NEW' (신규 지불수단 연동) 판별
2. R&R 검사: RNR-01~03 매칭 시 → REJECT + 이관 안내 + 대화 종료
3. 통과 시 22항목 체크:
   FOR each CHK-01..22:
       IF 요청서에 해당 항목 정의 없음:
           추가확인 항목으로 승격 (RAISE E-GOV-010)
           IF CHK.critical == true: 최우선 보완 요구
4. maps_to가 있는 항목은 해당 정책의 상세 검증(P-402/404/405/501 등)으로 위임
```

## 5. 실패 처리 및 에러코드

| 에러코드 | 의미 | 처리 |
|---|---|---|
| `E-GOV-001` | 디자인/퍼블리싱 영역 요청 | 반려 + 디자인팀 이관 안내 |
| `E-GOV-002` | 원천사 자체 시스템 영역 요청 | 반려 + 제휴 마케팅팀 이관 안내 |
| `E-GOV-003` | 단순 설정/운영 업무 요청 | 반려 + 운영/정산 파트 이관 안내 |
| `E-GOV-010` | 22항목 중 누락 발생 | 추가확인 항목 승격 + 대화로 보완 요구 |

## 6. 변경 이력 및 과거 누락 이슈(History)

### 6-1. 과거 개발건 주요 누락 이슈
- **사례 A (카카오페이 연동):** '부분취소 불가' 제휴사 정책을 사전 인지하지 못하고 통합 결제창에 연동 → 오픈 후 환불 민원 시 전체취소 후 재결제 운영 공수 발생. *(대책: CHK-16 초기 필수 확인)*
- **사례 B (네이버페이 다이렉트):** 다이렉트 호출(CHK-04)을 원했으나 인증 주체(CHK-06) 스펙 협의 누락 → 개발 도중 API 구조 전면 수정, 일정 2주 지연.
