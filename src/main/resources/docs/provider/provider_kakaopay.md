# [원천사 규격] Kakaopay 온라인 단건결제 표준 프로세스

카카오페이 단건결제는 가맹점 서버가 카카오페이 결제 서버와 직접 통신하는 [결제 준비 -> 결제 요청(사용자 인증) -> 결제 승인]의 3단계 흐름으로 진행됩니다.

## 1.1 단계별 API 연동 스펙 및 체크리스트

### [Step 1] 결제 준비 API (`POST /v1/payment/ready`)
- **설명:** 가맹점이 주문 정보를 카카오페이 서버에 전달하고, 결제 고유 번호(`tid`)와 카카오톡 결제창으로 이동할 리다이렉트 URL을 발급받는 단계입니다.
- **★ 현업 확인 필수 사양 (Edge Case):**
  - **과세/비과세 금액 분리:** 카카오페이는 요청 전문에 총액(`total_amount`)과 함께 비과세 금액(`tax_free_amount`), 부가세 금액(`vat_amount`)을 명확히 구분하여 전달해야 합니다. 현업 담당자는 복합과세 대상 상품(예: 면세 품목 포함)인지 여부를 명시해야 합니다.
  - **리다이렉트 URL 처리:** 결제 성공(`approval_url`), 취소(`cancel_url`), 실패(`fail_url`) 시 사용자가 복귀할 각각의 화면 웹뷰 동선이 기획서에 포함되어야 합니다.

### [Step 2] 결제 요청 및 사용자 인증
- **설명:** 발급받은 리다이렉트 URL(`next_redirect_app_url` 또는 `next_redirect_pc_url`)을 이용해 고객에게 카카오톡 결제 화면을 제공하고 인증을 유도합니다.
- **★ 현업 확인 필수 사양 (Edge Case):**
  - **모바일 웹/앱 전환 (App-to-App):** 인앱 웹뷰(WebView) 환경에서 카카오톡 앱을 호출할 때 Android/iOS 각각의 인텐트 스키마(Intent Schema) 처리가 필요하므로 앱 개발팀의 공수 조율이 필수적입니다.

### [Step 3] 결제 승인 API (`POST /v1/payment/approve`)
- **설명:** 사용자가 인증을 완료하면 카카오페이가 가맹점에 전달하는 `pg_token`과 준비 단계에서 받은 `tid`를 사용하여 최종 승인을 요청하는 단계입니다.
- **★ 현업 확인 필수 사양 (Edge Case):**
  - **TID 관리 및 매핑:** 결제 준비 단계에서 발급받은 `tid`는 세션(Session)이나 DB에 안전하게 보관되어야 승인 단계에서 대조할 수 있습니다. 현업이 결제창 진입 후 이탈했다가 재진입하는 시나리오를 기획할 경우 `tid` 채번 정책을 재정의해야 합니다.

## 1.2 비즈니스 정책 가이드
1. **테스트 가맹점 코드(CID):** 개발 테스트 단계에서는 `TC0ONETIME`이라는 테스트 전용 CID를 사용하지만, 운영 서버 배포를 위해서는 카카오페이와 사전 가맹 계약을 통해 고유 CID를 발급받아야 하므로 오픈 일정을 역산해야 합니다.
2. **결제 수단 제어:** 현업이 특정 프로모션으로 인해 "카카오페이 카드 결제만 막고 머니 결제만 유도하고 싶다"고 요청할 경우, 준비 API의 `available_cards` 파라미터 제어나 결제 수단 필터링 스펙이 기획서에 명시되어야 합니다.
