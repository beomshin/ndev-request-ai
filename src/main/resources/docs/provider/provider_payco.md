# [원천사 규격] PAYCO 온라인 간편결제 표준 흐름 (Flow)

인증형 간편결제는 가맹점 백엔드(Server)와 PAYCO 결제 서버 간의 [주문예약 -> 결제창 호출 -> 결제승인] 3단계 구조로 진행됩니다. 현업 담당자는 결제창만 뜨면 끝나는 것이 아니라, 백엔드에서 사전 예약과 사후 승인 단계가 필수적임을 인지해야 합니다.

## 1.1 단계별 연동 스펙 및 체크리스트

### [Step 1] 주문 예약 요청 (가맹점 Server -> PAYCO Server)
- **설명:** 고객이 결제 버튼을 누르면 가맹점 서버가 PAYCO에 주문 정보를 먼저 전달하고 `reserveOrderNo`(주문예약번호)를 받아옵니다.
- **★ 현업 확인 필수 사양 (Edge Case):**
  - **주문 이중화 방지:** 주문 예약 시 가맹점의 고유 주문번호(`sellerOrderReferenceKey`)가 중복되어 인입되면 PAYCO 시스템에서 에러를 반환합니다. 따라서 현업은 사용자가 결제창을 새로고침하거나 중복 클릭했을 때 주문번호가 새로 채번되거나 차단되는 UI 스펙을 기획에 반영해야 합니다.

### [Step 2] 결제창 호출 및 인증 (가맹점 WEB/APP -> PAYCO 모바일 결제창)
- **설명:** 받아온 주문예약번호를 가지고 고객에게 PAYCO 결제 화면을 띄워 인증을 진행합니다.
- **★ 현업 확인 필수 사양 (Edge Case):**
  - **앱카드 App-to-App 호출:** 모바일 앱 환경(WebView)에서 페이코 결제를 호출할 경우, 페이코 앱을 실행하기 위한 **Custom Scheme(App Scheme) 처리**가 앱 소스코드에 반영되어야 합니다. (현업이 앱 배포 일정을 잡을 때 반드시 앱 개발팀의 공수가 추가됨을 인지해야 함)

### [Step 3] 결제 승인 요청 (가맹점 Server -> PAYCO Server)
- **설명:** 고객이 인증을 완료하면 가맹점 백엔드가 최종적으로 PAYCO에 승인 API(`payment/approve`)를 호출하여 결제를 완료합니다.
- **★ 현업 확인 필수 사양 (Edge Case):**
  - **승인 타임아웃 처리:** 고객이 결제창에서 인증을 완료했으나 가맹점 서버의 일시적인 네트워크 순단으로 승인 요청이 누락되는 경우가 발생할 수 있습니다. 이를 방지하기 위한 망취소(Cancel) 로직이나 자동 배치 재검증 스펙이 포함되어야 합니다.

## 1.2 비즈니스 정책 가이드
1. **최소 결제 금액 제한:** PAYCO API는 1원 단위 승인이 가능하나, 가맹점 내부 정산 정책 및 수수료율에 따라 간편결제 최소 결제 금액(예: 1,000원 이하 제한)을 설정할 것인지 기획서에 명시해야 합니다.
2. **에러 코드 핸들링:** PAYCO 결제창을 닫거나(인증 취소), 한도 초과 등으로 결제가 실패했을 때 가맹점 자체 주문서 페이지로 자연스럽게 복귀(Redirect URL)시키는 동선 처리가 기획에 포함되어야 합니다.
