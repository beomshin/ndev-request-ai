# [원천사 규격] Toss Payments 기능별 연동 가이드 및 API 레퍼런스

본 문서는 토스페이먼츠가 제공하는 주요 결제 아키텍처(단건, 빌링, 브랜드페이)의 개념과 필수 체크리스트입니다. AI는 현업의 요구사항 성격에 따라 해당 가이드를 매핑하여 인터뷰를 리드해야 합니다.

## 1.1 기능별 연동 사양서 요약 (Guides v2)

* **자동결제(빌링) 이해하기** (`guides/v2/billing.md`)
  - **요약:** 정기 배송, 스트리밍 구독 등에서 사용되는 결제 방식. 카드/계좌를 최초 1회 등록해 빌링키를 발급받은 후, 이후에는 고객 인증 없이 백엔드 호출만으로 결제를 청구하는 흐름의 개념 제시.
* **자동결제(빌링) API로 연동하기** (`guides/v2/billing/integration-api.md`)
  - **요약:** 결제창 UI를 띄우지 않고, 가맹점 자체 화면에서 카드 정보를 직접 입력받아 빌링키를 발급받는 Server-to-Server(서버 투 서버) 연동 방식 사양.
* **퀵계좌이체 자동결제(빌링) 연동하기** (`guides/v2/billing/integration-quick.md`)
  - **요약:** 고객 계좌를 최초 1회 등록한 뒤, 비밀번호나 생체인증 등의 복잡한 절차 없이 가맹점의 결제 요청만으로 출금 처리하는 간편 계좌 빌링 연동 규격.
* **자동결제(빌링) 결제창 연동하기** (`guides/v2/billing/integration.md`)
  - **요약:** 토스페이먼츠가 제공하는 표준 카드 등록창(UI)을 띄워 고객이 직접 카드를 안전하게 등록하고 빌링키를 채번하는 클라이언트 연동 가이드.
* **브랜드페이 이해하기** (`guides/v2/brandpay.md`)
  - **요약:** 쿠팡의 쿠페이, 쓱페이처럼 가맹점 고유의 브랜드명을 내건 '자체 간편결제 시스템'을 구축하는 스펙. 회원 가입, 결제수단 등록/관리, 자체 비밀번호 인증 플로우 포함.

## 1.2 단계별 API 연동 스펙 및 예외 처리 (Edge Case)

### [Step 1] 결제 승인 API (`POST /v1/payments/confirm`)
- **결제 검증 키 대조:** 승인 요청 시 `paymentKey`, `orderId`, `amount` 3가지가 인증 결과와 일치하는지 가맹점 백엔드에서 밸리데이션 필수.
- **멱등성(Idempotency-Key) 보장:** 네트워크 재시도 시 중복 결제를 방지하기 위해 HTTP 헤더에 `Idempotency-Key` 설정을 강력히 권장.

### [Step 2] 자동결제(빌링) 청구 API
- **빌링키 유효성 만료:** 카드 분실, 재발급, 유효기간 만료 등으로 인해 정기결제 실패 시(`ERR_INVALID_CARD`) 고객에게 알림톡을 발송하여 결제 수단을 갱신하게 만드는 기획 동선 필요.
- **정기결제 예정일 배치 정책:** 매월 정산일에 대량의 배치(Batch) 프로세스가 돌 때, 실패 건에 대한 재시도(Retry) 주기 정책 요구사항 명시 필요.

### [Step 3] 망취소(Net Cancel) 예외 처리
- 가맹점 서버가 승인 API를 호출했으나 타임아웃이 발생한 경우, 거래대사 불일치 방지를 위해 즉시 망취소 API를 호출하는 비즈니스 로직을 구축해야 함.
